Architecture

Confirmed Tech Stack

Three mandatory layers with strict separation — no cross-layer direct calls. Each layer is independently deployable and testable.

Layer 1 · Messaging
Twilio SMS/MMS + iMessage

Twilio for SMS/MMS (Android + basic phones). SendBlue for iMessage to iPhone users — blue bubbles, auto SMS fallback built-in. Both channels abstracted behind a single MessagingService interface.

Layer 2 · AI
Claude API (Anthropic)

claude-haiku-4-5 default. Coaching, vision (MMS nutrition), crisis classification, session summarisation. No Twilio or DB calls.

Layer 3 · Data
PostgreSQL 15 + Redis 7

PostgreSQL: all durable storage. Redis: session cache (4h TTL) + BullMQ queues. No messaging or AI logic here.

Backend
NestJS 10 · TypeScript 5

Node.js 20 LTS. Modular architecture maps directly to the 3-layer structure. BullMQ for all async processing.

Frontend
Next.js 14 · App Router

Landing page + multi-step onboarding form only. Stripe Payment Element for card capture. Deployed on Vercel.

Payment
Stripe

SetupIntent → Subscription (30-day free trial). Only Stripe customer ID + subscription ID stored. Raw card data never touches RYKE servers.

Crisis Detection
Claude API + BERT Hybrid

Claude API classifier (primary) + Transformers.js local BERT (fast-path/fallback). Confidence threshold 0.65. Fully async via BullMQ.

Infrastructure
AWS EC2 + RDS + ElastiCache

Backend on EC2, PostgreSQL on RDS, Redis on ElastiCache. Docker Compose for local dev. Horizontal scaling of message workers without code changes.

Testing
Jest + Supertest

Unit, integration, and contract tests. Contract tests required for Twilio, Claude, Stripe, and PostgreSQL before any integration work.

Git Branching Strategy

Feature-branch workflow with mandatory PR reviews and constitution checks. No direct commits to main.

main → Production-ready code only. Protected branch. Auto-deploys to production on merge.
001-sms-ai-coaching → Current feature branch. All Phase 1 MVP development happens here. Merges to main on feature completion.
001-sms-ai-coaching/task-xxx → Individual task branches cut from the feature branch. One branch per task from tasks.md. PR back to feature branch.
hotfix/issue-description → Critical production fixes only. Cut from main, merged back to main AND feature branch.

Quality Assurance

Testing Strategy

Tests are written before implementation. A feature is not done until all predefined scenarios pass in CI. Four testing layers, each with a clear responsibility.

Unit Tests

Individual service methods in isolation. Mock all external dependencies. Fast — runs on every commit.

npm run test
Integration Tests

Full request flows against real PostgreSQL + Redis (Docker). No mocked DB. Covers onboarding, coaching, MMS, crisis flows.

npm run test:integration
Contract Tests

Verify external API contracts: Twilio webhook format, Claude API message shape, Stripe event structure, PostgreSQL schema.

npm run test:contract
E2E Tests

Real SMS sent to Twilio number on staging. Verifies full onboarding flow, coaching loop, MMS nutrition, and crisis detection on real devices.

Constitution Enforcement

10 Quality Gates

Every PR must include a constitution check section confirming all applicable gates pass. Non-negotiable gates cannot be bypassed under any circumstance.

📱
SMS-First Gate
Every coaching output must be deliverable as SMS. Flag any feature requiring a browser or app (onboarding form is the sole exception).
🛡️
Safety Gate NON-NEGOTIABLE
Mental health or distress features MUST include handoff logic + integration test covering the handoff path.
🔒
Privacy Gate NON-NEGOTIABLE
Features handling health or personal data MUST include encryption, access control, and consent mechanism.
🔌
Extensibility Gate
New health metric entities MUST use source-agnostic schema. No device-specific fields at the top level.
📊
Observability Gate
Every new service operation MUST emit a structured log event. Token usage logged per turn.
🎯
MVP Scope Gate NON-NEGOTIABLE
Phase 2+ features MUST be blocked and deferred to backlog. Any PR referencing out-of-scope items is rejected.
🧪
Test-First Gate
PRs MUST include predefined test files matching the spec's acceptance scenarios. No test = no merge.
🪙
Token-Efficiency Gate
New AI prompt templates MUST include estimated token count. Prompts exceeding 500 tokens/turn must be reviewed and justified.
📐
Separation of Concerns Gate
No cross-layer direct calls. Messaging / AI / Data layers must remain independently deployable and testable.
⚡
Zero-Friction Gate NON-NEGOTIABLE
Any onboarding change must be tested end-to-end: form submission → welcome SMS received in under 30 seconds.
