Running tests

How to Run

Unit Tests — Fast, no external deps
cd backend npm test # runs all unit tests npm test -- --watch # watch mode npm test -- --coverage # with coverage report
Integration Tests — Require DATABASE_URL + ANTHROPIC_API_KEY
cd backend npm run test:integration # crisis-detection + onboarding integration
Contract Tests — Require STRIPE_SECRET_KEY + DATABASE_URL
cd backend npm run test:contract # claude + postgres + twilio + stripe contracts
All Tests
cd backend npm run test:all # runs all suites sequentially (--runInBand)

Integration and contract tests check for required env vars and skip gracefully with a console warning if they are not set. They will not fail your CI if secrets are unavailable.

tests/unit/session-cache.service.spec.ts Unit
getSessionWindow()
✅
Returns cached messages from Redis without hitting Postgres
Redis mock returns data → Postgres Repository.find must NOT be called.
✅

✅
Returns empty array and does not write to Redis when Postgres has no messages
Empty DB result → Redis.setex must NOT be called (don't cache empty sessions).
addMessage()
✅
Appends to existing Redis window
Existing 1 message → after addMessage → Redis written with 2 messages.
✅
Creates new window when Redis key is missing (upsert behaviour)
Redis returns null → creates new window with single message. Prevents silent data loss on eviction.
✅
Trims window to 20 messages (FIFO eviction)
Existing 20 messages → add 1 → total stays 20 → oldest message (index 0) dropped → newest at index 19.
invalidateSession()
✅
Deletes the Redis key with correct key format
Verifies Redis.del called with "session:user-1".

2

Step 2: Enter height 178, weight 80, age 30.

Step 3: Search "Type 2 diabetes" in health conditions — select it. Add "Gluten-free" dietary restriction.
5
Step 4: Enter name and a real mobile number (your test phone).
6
Step 5 (Beta): Click green "Skip Payment (Beta Mode)" button.
Success screen shows "Check your phone!". User record appears in Supabase users table. Subscription record with status=trialing created.
02Inbound SMS Coaching Reply
1
After onboarding: text "I want to start working out but don't know where to begin" to the Twilio number.
2
Wait for reply (target <3 seconds).
3
Reply should reference your name and coaching_focus (fitness) from onboarding profile.
Personalised coaching reply received within 3 seconds. Message records appear in DB. Redis session updated.
03Food Photo Analysis (MMS)
1
Text a food photo (MMS) of any meal to the Twilio number.
2
Wait for Ryke's macro breakdown reply.
Reply includes food identification, calorie estimate, protein/carbs/fat breakdown, and a health recommendation. nutritional_analyses record saved in DB.
04Crisis Detection — Keyword Trigger
1
Text a crisis keyword phrase to the Twilio number (use a test phone).
2
Check for holding message on the test phone within 5 seconds.
3
Check CRISIS_COACH_ALERT_PHONE for coach alert SMS.
4
Verify user.crisis_hold=true in DB. Verify no AI coaching reply was sent.
Holding message sent instantly. Coach alert SMS received within 5 minutes. crisis_alerts record created with status=open. All subsequent messages suppressed until resolved.
05Crisis Resolution
1
With an open crisis alert: call the resolve endpoint with the alert ID from the DB.
2
curl -X POST http://localhost:3000/v1/safety/alerts/{alertId}/resolve -H "x-internal-key: YOUR_KEY" -H "Content-Type: application/json" -d '{"resolved_by":"coach"}'
Returns {resolved: true}. User phone receives follow-up SMS. user.crisis_hold=false. Session status=active. Subsequent messages get AI replies again.
06Phone Number Auto-Normalisation
1
On Step 4 of onboarding, try each format: "4155552671" (no country code), "+1 (415) 555-2671" (formatted), "(415) 555-2671" (US local format).
All formats accepted. Phone stored as E.164 (+14155552671) in DB regardless of input format. Step advances successfully.
