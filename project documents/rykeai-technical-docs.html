System design

Architecture Layers

The backend uses a 3-layer async architecture. Every inbound SMS flows through the messaging layer into BullMQ queues, processed by the AI layer, with all state persisted through the data layer.

📱
Inbound Gateway
Twilio webhook (SMS/MMS) · SendBlue webhook (iMessage) · Guards validate signatures
📋
Messaging Layer
MessagingProcessor (queue entry) → CoachingProcessor (AI pipeline) · BullMQ with retry + priority
🧠
AI Layer
CoachingService · CrisisService · VisionService · SummarisationService — all via Claude API
💾
Data Layer
PostgreSQL (TypeORM) for persistent state · Redis (ioredis) for session cache (20-msg sliding window, 4h TTL)
📤
Outbound Gateway
MessagingService → SendBlue (iMessage first) → Twilio SMS fallback · SafetyProcessor for coach alerts
💳
Billing Layer
StripeService · OnboardingService (DB transaction + Stripe compensation) · StripeWebhookController

Message Processing Flow

Inbound SMS
  → TwilioWebhookGuard (HMAC-SHA1 signature check)
  → MessagingController POST /webhooks/sms
  → BullMQ: messaging queue → send-message job
  → CoachingProcessor.process-coaching-message
      ├── Lookup user by phone_number
      ├── Update last_active_at
      ├── Save inbound Message to DB
      ├── SSRF-validated media URL check (Twilio domains only)
      ├── VisionService.analyzeFood() if NumMedia > 0
      ├── CrisisService.classify() ← ALWAYS AWAITED
      │     ├── Keyword fast-path (returns immediately)
      │     └── Claude ML classifier (fallback)
      ├── IF crisis: SafetyService.handleCrisisDetection() → RETURN (no AI reply)
      ├── SessionBoundaryService (check 4hr timeout)
      ├── SessionCacheService.getSessionWindow() ← Redis first, Postgres fallback
      ├── CoachingService.generateReply() ← Claude Haiku
      ├── SessionCacheService.addMessage() ← Update Redis window
      ├── Save AI Message to DB
      ├── SummarisationService (if 20 msgs or session expired)
      └── MessagingService.send() → SendBlue / Twilio

Other Tables

conversation_sessions
Tracks session lifecycle per user. Status: active | completed | crisis_hold. Tracks message_count and summary_generated flag. 4-hour inactivity timeout managed by SessionBoundaryService.
session_summaries
Claude-generated summaries stored when session hits 20 messages, 4hr timeout, or token budget. trigger: session_expiry | message_count | token_budget.
nutritional_analyses
Per-MMS food photo analysis: detected_foods[], calories, protein/carbs/fat grams, health_flags[], recommendation. UNIQUE on message_id.
processed_stripe_events
Idempotency table. stripe_event_id is PK. INSERT conflict on duplicate delivery → skip silently. Prevents double-processing webhooks.
Business logic

Core Services

🧠CoachingService
Generates AI coaching replies. Builds context-aware system prompt from user profile + session history + latest summary. Calls Claude Haiku (max_tokens=256). Returns reply text + token count. Logs token usage.

ai/coaching.service.ts
🛡️CrisisService
Two-stage classifier. Stage 1: keyword fast-path (instant, no API). Stage 2: Claude ML classifier with JSON response. Threshold: CRISIS_CONFIDENCE_THRESHOLD (default 0.65). Fail-safe: API error → crisis=true. Returns CrisisResult.

ai/crisis.service.ts
📸VisionService
Food photo analysis via Claude Vision. Validates media URL against TRUSTED_MEDIA_HOSTS (SSRF protection). Returns NutritionResult: food_identified, detected_foods, calories, macros, health_flags, recommendation.

ai/vision.service.ts
📝SummarisationService
Condenses session message history to a text summary. Triggered at 20 messages, 4hr timeout, or token budget. Saves to session_summaries. Updates session.summary_generated flag.

ai/summarisation.service.ts
💾SessionCacheService
Redis sliding window for session context. Max 20 messages, 4hr TTL. Cache miss → load from Postgres + write back. addMessage() upserts (creates key if evicted). invalidateSession() deletes key.

data/session-cache.service.ts
💬MessagingService
Outbound SMS routing. Tries SendBlue first (iMessage for iPhones). Falls back to Twilio SMS. Queues via BullMQ messaging queue with 3 retries + exponential backoff.

messaging/messaging.service.ts
🚨SafetyService
Handles confirmed crisis events. Sets crisis_hold, freezes session, sends holding message, creates CrisisAlert record, queues coach alert job. resolveAlert() restores normal flow.

safety/safety.service.ts
💳StripeService
Wraps Stripe SDK. createCustomer, createSetupIntent, createSubscriptionWithTrial (attach PM → set default → create sub), cancelSubscription, deleteCustomer, constructWebhookEvent.

onboarding/stripe.service.ts
Async processing

BullMQ Queues

All async work is dispatched via BullMQ backed by Redis. Queues survive restarts and retry on failure.

coaching

process-coaching-message

Main AI pipeline. Takes {from, body, twilioSid, numMedia, mediaUrl, mediaContentType}. Crisis check → optional vision → coaching reply → session update → summarisation. Priority 5, 3 attempts, exponential backoff.

messaging

send-message

Outbound SMS dispatch. Takes {to, body, type}. Routes to SendBlue or Twilio. 3 retry attempts with exponential backoff. Used for all outbound messages including welcome, billing, and coach-resolved follow-up.

crisis-detection

dispatch-coach-alert

Coach alert dispatch. Priority 10 (highest). Takes {alertId, userId, createdAt}. Sends SMS to CRISIS_COACH_ALERT_PHONE + email to CRISIS_COACH_ALERT_EMAIL. Logs SLA breach if >5 minutes. 3 attempts.

Security

Security Controls

🔐 Webhook Signature Validation
TwilioWebhookGuard: HMAC-SHA1 using APP_BASE_URL (not request.protocol — proxy-safe). SendBlueWebhookGuard: validates both key ID and secret using crypto.timingSafeEqual (prevents timing attacks).
🛡️ Internal API Key Guard
InternalApiKeyGuard protects safety resolution, data export, and GDPR delete endpoints. Key must be ≥32 chars (set via INTERNAL_API_KEY). Constant-time comparison.
🚫 SSRF Protection
Media URLs from MMS messages are validated against TRUSTED_MEDIA_HOSTS: ['api.twilio.com', 'media.twiliocdn.com']. Non-matching URLs are rejected before Claude Vision call.
🔒 Input Validation
All DTOs use class-validator + class-transformer. Global ValidationPipe with whitelist:true (strips unknown fields). Joi schema validates all env vars at startup (abort-early:false).
🌐 HTTP Security
Helmet middleware (security headers). ThrottlerModule: 60 req/min default, 10 req/min strict (onboarding). CORS restricted to FRONTEND_URL. Trust proxy enabled for Render/Railway deployment.
🏦 Stripe Idempotency
ProcessedStripeEvent table prevents double-processing. INSERT + unique constraint catch (code 23505) pattern — no read-before-write race condition. Stripe webhook signature verified on every event.
🛟 Crisis Fail-Safe
If Claude API is unavailable during crisis classification, the system returns crisis=true (conservative hold). Never silently fail. Coach is alerted even on classifier error.
💳 Payment Safety
OnboardingService uses DB transaction. If Postgres write fails after Stripe objects are created, compensating actions cancel the Stripe subscription and delete the customer to prevent orphaned billing.

Development

Local Setup & Commands

# Install dependencies
cd backend && npm install
cd frontend && npm install

# Run database migrations (against configured DATABASE_URL)
cd backend && npm run migration:run

# Start backend (watch mode — auto-reloads on file changes)
cd backend && npm run start:dev   # http://localhost:3000

# Start frontend
cd frontend && npm run dev        # http://localhost:3001

# Type check
cd frontend && npx tsc --noEmit

# Run unit tests
cd backend && npm test

# Run all tests (unit + integration + contract)
cd backend && npm run test:all
# Health check
curl http://localhost:3000/v1/health
# → {"status":"ok","checks":{"postgres":"ok","redis":"ok"}}
